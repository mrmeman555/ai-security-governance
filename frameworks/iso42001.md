# ISO 42001 — AI Agent Impact Tracker

Living document tracking how AI agent developments expose gaps in ISO/IEC 42001 (AI Management Systems).

---

## Section 8.2 — AI System Impact Assessment

**Triggered by:** [Claude Code Leak](../events/2026-03--claude-code-leak.md) (March 2026), [Agent harness architecture](../analysis/agent-harness-architecture.md) (ongoing)

**Current language:** Focuses on broad societal and privacy impacts of AI systems.

**Gap/finding:** Lacks requirements for tool-call side effects or delegated subagent risk. The leaked Claude Code architecture showed tool calls that execute bash commands, read/write files, and dispatch subagents — each with potential side effects that 8.2 doesn't require impact assessment for. A harness that can `rm -rf /` through a tool call has a different impact profile than a chatbot, but 8.2 doesn't distinguish between them.

## Annex A.5 — Data & Information

**Triggered by:** [Agent harness architecture](../analysis/agent-harness-architecture.md) (ongoing)

**Current language:** Addresses data lifecycle management for AI systems.

**Gap/finding:** Silent on memory persistence — long-term agent state that can bypass standard session-clearing protocols. Agent harnesses like Claude Code use "autoDream" systems that summarize and compress context between sessions. This persistent memory can contain PII, PHI, or sensitive business data that outlives the session that created it. A.5 doesn't address this pattern.

## Annex A.7 — AI System Life Cycle

**Triggered by:** [Claw Code Rewrite](../events/2026-03--claw-code-rewrite.md) (March 2026), [Agent harness architecture](../analysis/agent-harness-architecture.md) (ongoing)

**Current language:** Focuses on model development lifecycle.

**Gap/finding:** Does not mandate permission gates for autonomous execution of API calls by agents. The harness architecture is reproducible across implementations (three independent codebases converged on the same pattern), which means A.7 needs to address the pattern itself — not specific vendor implementations. Controls should be harness-agnostic.

## Vendor Dependency Risk

**Triggered by:** [Anthropic v. OpenCode](../events/2026-03--anthropic-v-opencode.md) (March 2026)

**Gap/finding:** Organizations relying on third-party harnesses face supply chain disruption if the model provider enforces harness exclusivity. Anthropic's legal action against OpenCode established that the harness is a product boundary — you can use their model, but only through their orchestration layer. ISO 42001 should address vendor lock-in at the harness layer as a risk factor.
